# Требования к сервису бронирования

## Функциональные требования

1. Пользователь может создать событие совместного просмотра фильма.
2. Пользователь указывает фильм, дату и время, место, лимит мест и описание.
3. Пользователь может посмотреть список активных событий.
4. Пользователь может посмотреть подробную информацию по одному событию.
5. Авторизованный пользователь может забронировать место на событие.
6. Пользователь не может забронировать одно и то же событие повторно.
7. Система не позволяет забронировать больше мест, чем указано в лимите события.
8. Пользователь может отменить свою бронь.
9. Пользователь может посмотреть список своих бронирований.
10. Владелец события может отменить созданное событие.

## Нефункциональные требования

1. API должен отдавать ответы быстрее 300 мс на типовых запросах.
2. Ошибки пользователя должны возвращаться как 4xx, а не как 5xx.
3. Если событие не найдено, сервис возвращает 404.
4. Если пользователь пытается забронировать заполненное событие, сервис возвращает 409.
5. Если пользователь пытается отменить чужое событие, сервис возвращает 403.
6. Бронирование должно быть защищено от гонки данных через транзакцию и блокировку строки события.
7. Сервис должен запускаться локально через Docker Compose.
8. Структура проекта должна быть понятной и похожей на сервисы курса.
9. Сервис должен хранить только необходимые данные: `user_id`, `movie_id`, данные события и бронирования.
10. Сервис должен быть готов к дальнейшему расширению: уведомления, оплата, рейтинг участников.

## Ограничения MVP

1. Оплата не реализуется.
2. Фронтенд не реализуется.
3. Уведомления не реализуются.
4. Сервис фильмов используется как внешняя зависимость.
5. Auth используется через JWT-токен.
